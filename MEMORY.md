## 认知突破

### Orchestra 长程任务框架（2026-03-06）

**问题：** 如何在脱离用户交互的情况下，实现小时级自主长程工作？

**解决方案：** 分段执行 + 状态持久化 + 断点恢复

```
时间线: Segment 1 → Segment 2 → Segment 3 → Report
状态: queue.json 持久化，支持断点恢复
```

**核心组件：**
- `workspace/orchestra/queue.json` - 任务队列和状态
- `workspace/orchestra/agents.json` - Agent 定义和配置
- `skills/orchestra/SKILL.md` - 编排器 Skill

**试运行结果：**
- 5 个任务全部完成
- 执行时间 ~20 分钟
- 生成了完整的 AI Agent 技术侦察报告

**关键洞察：**
- 不依赖 sessions_spawn，通过状态文件实现协作
- 每个 segment 可由独立 Cron 任务触发
- 支持断点恢复，适合长时间运行

## 认知突破

### NightOwl v2.0 主理人架构（2026-03-06）

**问题：** AI agent 会话 30 分钟无活动会超时，无法实现真正的夜间自主工作。

**核心洞察：** 用户需要的不是"定时任务"，而是一个**可信的夜班同事**——睡前说一句"晚安，去吧"，醒来时世界已经不一样了。

**解决方案：** 主理人（Orchestrator）模式

```
用户说"晚安，去吧"
       ↓
主理人进程启动（独立 Node.js 脚本）
       ↓
┌─────────────────────────────────────┐
│  主理人职责（持续 3 小时）             │
│  - 执行探索任务                      │
│  - 状态实时写入文件                   │
│  - 完成后自动发送钉钉报告             │
│  - 全程无需用户参与                   │
└─────────────────────────────────────┘
       ↓
用户醒来收到报告
```

**关键设计：**
- 不依赖 OpenClaw 会话保持
- 状态文件持久化（nightowl-state.json）
- 钉钉通知作为唯一用户交互点
- 支持启动/停止/状态查询

**文件位置：** `workspace/nightowl/`

### Web API 公网暴露（2026-03-06）

**需求：** 让其他人公网调用 OpenClaw 能力。

**方案：** Express + Cloudflare Tunnel

```
公网用户
   ↓
Cloudflare Tunnel（免费，无需账号）
   ↓
Express 服务器 (localhost:3456)
   ↓
OpenClaw 工具调用
```

**公网地址：** `https://*.trycloudflare.com`（临时，每次重启变化）

**接口：**
- `POST /api/execute` - 执行工具
- `POST /api/nightowl/start` - 启动 NightOwl
- `GET /api/nightowl/status` - 查看状态

**生产建议：**
- 启用 API Key 认证
- 使用 Cloudflare 命名隧道（固定域名）
- 添加限流保护

**文件位置：** `workspace/web-api/`

# 记忆可视化项目规划

**项目名称**：Memory Vision / 记忆之眼
**创建时间**：2026-03-04
**提出者**：顾丽红
**目标**：将文本记忆可视化，提高信息传递效率

---

## 一、需求分析

### 1.1 核心问题
- 当前记忆系统是纯文本（Markdown）
- 人类对图像的理解速度远快于文字
- 难以发现记忆之间的关联和模式

### 1.2 用户场景

| 场景 | 用户需求 | 可视化方案 |
|------|---------|-----------|
| 回顾一天 | 快速了解今天发生了什么 | 时间线视图 |
| 追踪任务 | 查看待办事项进度 | 看板/甘特图 |
| 学习成长 | 了解错误模式和成长轨迹 | 错误热力图 + 成长曲线 |
| 发现关联 | 发现事件/决策之间的联系 | 知识图谱 |
| 全局概览 | 一眼看懂整体状态 | Dashboard 仪表盘 |

### 1.3 数据源

```
OpenClaw/
├── MEMORY.md          → 长期记忆（关键决策、重要事件）
├── TODOS.md           → 任务追踪（状态、优先级）
├── MISTAKES.md        → 错误记录（分类、教训）
├── CAPABILITIES.md    → 能力边界（技能清单）
└── memory/
    └── YYYY-MM-DD.md  → 每日记录（时间序列数据）
```

## 二、技术方案

### 2.1 产品形态选择

| 方案 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| **Obsidian 插件** | 无缝集成、用户熟悉、易分发 | 受限于 Obsidian 能力 | ⭐⭐⭐⭐ |
| **独立 Web 应用** | 灵活、可定制、可独立部署 | 需要额外维护、用户需切换 | ⭐⭐⭐ |
| **OpenClaw Skill** | 与 OpenClaw 深度集成 | 依赖 OpenClaw 生态 | ⭐⭐⭐⭐⭐ |
| **CLI 工具** | 简单、快速、开发者友好 | 可视化能力有限 | ⭐⭐ |

**推荐方案**：OpenClaw Skill + Obsidian 插件双轨并行

### 2.2 技术栈

**前端：**
- 框架：React 18 + TypeScript
- 可视化：
  - D3.js（图谱、时间线）
  - React Flow（节点关系图）
  - ECharts（统计图表）
- UI 组件：Tailwind CSS + shadcn/ui

**后端（如需要）：**
- Node.js + Express / Hono
- 或直接静态站点 + Markdown 解析

**数据处理：**
- gray-matter（Frontmatter 解析）
- marked / remark（Markdown 解析）
- 自定义解析器（提取结构化数据）

### 2.3 架构设计

```
┌─────────────────────────────────────────────────────┐
│                    用户界面层                         │

│  └───────────┴───────────┴───────────┴───────────┘ │
└─────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────┐
│                   数据处理层                          │
│  ┌──────────────┐  ┌──────────────┐  ┌───────────┐ │
│  │ Markdown 解析 │  │ 数据标准化   │  │ 关系提取  │ │
│  └──────────────┘  └──────────────┘  └───────────┘ │
└─────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────┐

│  MEMORY.md │ TODOS.md │ MISTAKES.md │ memory/*.md  │

```

## 三、多 Agent 协作设计

### 3.1 Agent 角色定义

| Agent | 职责 | 技能要求 |
|-------|------|---------|
| **Architect** | 整体架构设计、技术选型、模块划分 | 系统架构、技术广度 |
| **Frontend** | UI 实现、组件开发、交互设计 | React、可视化库 |
| **Data** | Markdown 解析、数据结构化、API 设计 | 数据处理、Node.js |
| **Product** | 用户需求、功能优先级、体验优化 | 产品思维、UX |
| **QA** | 代码审查、Bug 发现、测试用例 | 测试、质量保障 |

### 3.2 协作流程

```
1. Architect 设计整体架构
        ↓
2. Product 定义功能优先级
        ↓
3. Data 设计数据结构和解析逻辑
        ↓
4. Frontend 实现 UI 和可视化
        ↓
5. QA 审查和测试
        ↓
6. 迭代优化
```

### 3.3 任务分配示例

**Architect Agent 任务：**
- 设计项目目录结构
- 选择技术栈
- 定义模块边界和接口

**Frontend Agent 任务：**
- 实现 Dashboard 布局
- 开发时间线组件
- 开发知识图谱组件

**Data Agent 任务：**
- 编写 Markdown 解析器
- 设计数据结构
- 实现数据转换管道

## 七、风险和挑战

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 多 Agent 配置复杂 | 可能无法实现 | 先单 Agent 完成 MVP，再优化 |
| 可视化技术学习曲线 | 开发时间延长 | 使用成熟库，减少自研 |
| 数据格式不统一 | 解析困难 | 先标准化数据格式 |
| 性能问题（大量数据） | 加载慢 | 分页、懒加载、缓存 |
